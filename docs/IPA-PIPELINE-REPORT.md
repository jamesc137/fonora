## 4. Browser compatibility concerns

| Concern | Notes |
|---------|-------|
| HTTP required | WASM and `fetch()` fail on `file://` |
| First load | ~18 MB download for eSpeak voice data |
| WASM memory | ~32 MB heap; may fail on older mobile Safari |
| Async UI | Translator shows loading state while WASM initializes |
| GPL | eSpeak NG is copyleft; document if distributing publicly |
| Roman input for CJK/Arabic | eSpeak spells roman text letter-by-letter unless native script is used |

Supported: Chrome, Firefox, Edge, Safari (desktop). Mobile Safari works but may be memory-limited.

## 6. Recommendations for future language expansion

1. **Native script input** ‚Äî Pass Japanese kana/kanji and Arabic script to eSpeak for better IPA quality.
2. **Language-specific normalize tweaks** ‚Äî Optional post-process rules per language without changing Fonora symbols.
3. **Glossary IPA overrides** ‚Äî Allow dictionary entries to specify IPA or phoneme strings, bypassing eSpeak for known exceptions.
4. **Upgrade path** ‚Äî If `espeak-ng` npm stalls, evaluate `@echogarden/espeak-ng-emscripten` (newer build, Aug 2025).
5. **Lazy language packs** ‚Äî Split WASM data by language if bundle size becomes a blocker.
6. **Retire legacy rules incrementally** ‚Äî Use Pronunciation Testing compare mode to validate IPA coverage, then deprecate `normalize.js` categories one at a time.
7. **Tone handling research** ‚Äî Fonora has no tone symbols; document intentional omission or plan separate suprasegmental layer later.
